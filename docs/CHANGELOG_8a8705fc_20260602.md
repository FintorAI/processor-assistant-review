## 3. Document Extraction — Fire-and-Poll on Cache Miss (`output/tools/data_gathering.py`)

**Problem:** `fetch_doc_fields` was GET-only. If DynamoDB cache was empty (as in thread `019e86ec`), all 31 required docs came back as `not_found` stubs and downstream review steps ran on empty data.

**Fix — `_sequential_extract_and_collect` helper + updated `fetch_doc_fields`:**

New flow:
```
GET /efolder → check which required doc types are missing from cache
  → For each missing type: POST /efolder/direct (trigger extraction)
  → Poll GET every 5s until all pending types resolve (max 120s)
  → Final GET with includeFields=true to collect results
  → Merge with cached docs → filter → normalize → state
```

Key behaviours:
- Only triggers extraction for doc types **not already in cache** (partial cache handled correctly)
- `force_extract: true` in state bypasses cache entirely and re-extracts all types
- `_meta.source` in `efolder_documents` records which path ran (`"efolderGet"` vs `"efolderExtract (POST+poll)"`)
- Doctype alias normalisation maps variant DynamoDB names back to canonical required names

## 4. New Tool — `wait_for_pending_docs` (`output/tools/data_gathering.py`)

**Problem:** If extraction times out during step 0, docs remain in `status: "pending"`. Previously there was no way to wait for them before a step that needs them — the agent would either skip or produce false "document missing" flags.

**Fix — `wait_for_pending_docs(doc_types: list[str])` tool:**
- Polls `GET /efolder` every 5s (up to 90s) until each requested type resolves
- On resolve: fetches full extracted fields, merges into both `efolder_documents` and `doc_fields` in state
- Returns `{resolved, failed, timed_out}` — always proceeds, never blocks the whole run
- `fetch_doc_fields` now explicitly signals pending types in its return message: _"EXTRACTION STILL IN PROGRESS for: [...]. Call wait_for_pending_docs([...]) before any step that needs these documents."_

**Registration:**
- Added to `GENERAL_TOOLS` in `output/registry.py` → available in **every step**
- Imported in `output/tools/__init__.py` and `get_all_tools()`
- Listed in `output/config/workflow_config.json` under `STEP_00.tools`

## 5. Bug Fix — Boolean Checkbox Write Format (`output/tools/review_borrower_summary.py`)

**Problem:** Fields 4920 (`Borrower Accept Text/SMS`) and 4935 (`Co-Borrower Accept Text/SMS`) are boolean checkboxes. Writing `"Y"` returns `400 Bad Request: Invalid field name or value (Serialization)`. Encompass only accepts `"true"` / `"false"` as write values for checkbox fields (reads come back as `Y`/`N`).

**Verified via `field_rw.py` on loan 2605926537 (test):**
- `"true"` → write succeeds, stored as `Y` ✓
- `"Y"` → 400 Serialization error ✗

**Fix:** Changed both writes from `"Y"` → `"true"`:
```python
# Before
{"4920": "Y" if not _is_checked(borrower_accept_sms) else None}
# After
{"4920": "true" if not _is_checked(borrower_accept_sms) else None}
```

**General rule:** All Encompass boolean/checkbox fields must be written as `"true"`/`"false"` (string), not `"Y"`/`"N"`.

## 23. Reverted `_efolder_present` `not_found` Guard + FACTORY-LOCK on `_helpers.py`

**Problem:** Entries #8/#16 added a `status == "not_found"` short-circuit to `_efolder_present`.
This was wrong. In Encompass, a document's eFolder bucket can hold an attachment (the document
IS present) while DynamoDB field extraction returns `status: "not_found"` with `copy_count: 1`.
The guard then reported the present document as **missing**, producing a false
"Missing Required Document: 1003 URLA" critical flag.

**Verification (loan 2605926537, Test — run via `local/scripts/inspect_efolder.py`):**
- `1003` bucket → has a 1224KB attachment, status `Received` → document **present**, but extraction
  returned `not_found`. Correct behaviour: NOT flagged missing.
- `Driver's License` → no `ID Customer Identification Documentation` bucket exists at all →
  genuinely **absent** → correctly produces the "Govt ID Not Yet Uploaded" info flag.

Both render identically in saved state (`status=not_found, copy_count=1`), so the status field
cannot distinguish present-but-unextracted from absent. The reliable signal is `copy_count`:
a truly-absent doc (no DynamoDB record) gets `copy_count=0` via the stub path, while a
present-but-unextracted doc keeps `copy_count>=1`.

**Fix:**
- Removed the `not_found` short-circuit from `_efolder_present` in both
  `output/tools/_helpers.py` and the template `factory/templates/tool_helpers.py.j2`.
  Restored the original `copy_count`-based presence check (matches working commit `c0a78a0`).
- Reverted an in-progress experiment in `data_gathering.py` that changed
  `_build_encompass_bucket_map` to return a tuple (the caller was never updated, leaving the
  module broken). Back to the single-dict return.

**FACTORY-LOCK on `_helpers.py`:** The repeated loss of `_helpers.py` edits (#8 → #16) was
because `factory.code_generator.generate_tool_helpers()` always regenerated the file from the
template, ignoring FACTORY-LOCK. Two changes:
- `generate_tool_helpers()` now checks the first 15 lines for `# FACTORY-LOCK: true` and skips
  regeneration if present (logs `Skipping … (FACTORY-LOCK: true)`).
- Added `# FACTORY-LOCK: true` to the `_helpers.py` module docstring.

`factory-reset` now logs `Skipping …/_helpers.py (FACTORY-LOCK: true)` and validation still passes.
The `_efolder_bucket` helper (added in #14-era work) is also now permanently safe.

## #25 — USPS DPV parsed from wrong object → valid address falsely flagged `Property Address Invalid`

**Problem:** `Property Address Invalid` (warning) fired for `2814 Hemlock Ave, Baltimore, MD 21214`
even though USPS recognizes it as a real, deliverable address (it returns full ZIP+4 `21214-1245`).
The earlier message fix (#10) only improved the wording (`DPV null`) — it didn't address the root
cause. `address_validation.valid` is gated on `dpv_confirmation in ("Y","S","D")`, and
`dpv_confirmation` was always `null`.

**Root cause:** USPS Addresses **v3** splits the response into two objects:

```
data["address"]        → standardized street/city/state/ZIP/ZIPPlus4
data["additionalInfo"] → DPVConfirmation, deliveryPoint, carrierRoute, vacant, DPVCMRA, business, ...
```

`shared/usps_validator.py` read `DPVConfirmation` (and `deliveryPoint`, `carrierRoute`, `vacant`,
`DPVCMRA`, `business`, `centralDeliveryPoint`) from `data["address"]`, where they never exist — so
DPV was always `None` and every address was treated as unconfirmed.

Live response for the flagged address:
```json
"additionalInfo": { "DPVConfirmation": "Y", "deliveryPoint": "14", "carrierRoute": "C013", "vacant": "N" }
```

**Fix:** Read all delivery flags from `data["additionalInfo"]` in `validate_address()`
(`shared/usps_validator.py`, hand-written — not factory-generated). After the fix the live call
returns `dpv_confirmation='Y'`, `valid=True`, so the false flag no longer fires for deliverable
addresses while genuinely undeliverable ones (DPV `N`) still flag correctly.

**Verified:** live USPS call for `2814 Hemlock Ave` now returns `DPV=Y`, `delivery_point=14`,
`carrier_route=C013`, `vacant=N`.
