- Same bank fix was not applied yet in 1003 URLA Part 3 2a
  - ✅ **Two-part fix, both done (2026-07-22) — and corrected to the real API contract.** Per the screenshots: when Encompass has 2+ accounts at the same institution, they should show up as **one VOD "Belong To" entry** with multiple rows in its Account Information grid (Checking + Savings both under one "Burke & Herbert Bank & Trust" row) — same as the user manually did in the screenshot.
  - **Corrected API contract (per the "V3 Manage VODs" reference the user pasted):** there is only **one PATCH endpoint** on the whole VODs *collection* — `PATCH /loans/{loanId}/applications/{applicationId}/vods` — no `{vodId}` in the path, no separate POST, and no HTTP DELETE verb. An `action` query param (`add` / `update` / `reorder` / `delete`) selects the operation, and the body is an array of VOD objects (matched by `id` for update/delete) processed as one transaction. All three VOD functions in `encompass_client.py` were rewritten to match this — my earlier versions had guessed at a POST-per-account and a per-`{vodId}` PATCH/DELETE, which was wrong:
    1. **Going forward (new accounts):** `add_vod_accounts()` groups accounts by `(institution_name, owner)`, then sends **one batched `PATCH ?action=add`** with an array of new VOD objects (one per institution) — instead of one POST per account.
    2. **Filling blanks on existing entries:** `update_vod_accounts()` now sends **one batched `PATCH ?action=update`** (body: `[{"id": vod_id, "items": [...]}, ...]`) for all VODs needing completion, instead of one PATCH per VOD hitting a `{vodId}` path that doesn't actually exist.
    3. **Existing duplicates (what you're pointing at):** `merge_duplicate_vods()` — new function — detects VOD entries that already exist as separate top-level rows for the same institution, then runs **one `PATCH ?action=update`** (combining every group's items into its first/kept entry) followed by **one `PATCH ?action=delete`** (removing all surplus entries in one call) — the delete only runs after the merge succeeds, so accounts are never lost even if the delete step fails.
  - ⚠️ **Not auto-run yet in 6.1.** `review_urla_assets.py` (6.1) still only *detects and flags* (`"Duplicate VOD Entries at Same Institution"`, warning) — the request shapes are now confirmed from the official API docs (much higher confidence than my earlier guess), but this hasn't been exercised against a live loan yet. Say the word and I'll wire 6.1 to call it for real.
  - Verified with mocked-network tests against the corrected contract: (a) 2 new accounts at the same bank + 1 at a different bank → exactly 1 `PATCH ?action=add` call with both VOD objects in the body array; (b) 2 pre-existing separate entries at the same bank → 1 `PATCH ?action=update` (merges both accounts into the first entry) + 1 `PATCH ?action=delete` (removes the second entry), a third entry at a different bank left untouched; (c) `update_vod_accounts` blank-fill → 1 `PATCH ?action=update`; (d) 6.1 correctly raises the warning flag when duplicate entries are present in `state["vod_data"]`.
