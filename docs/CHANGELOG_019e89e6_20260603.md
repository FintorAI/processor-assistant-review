## 1. Bank Statements reported "0 months" â€” schema name mismatch (plural vs singular)

**Symptom:** `run_pre_checks` (1.1) flagged _"Insufficient Bank Statements â€” Found 0 month(s); 2 required for Conventional"_ even though a Bank Statement was present in the eFolder (Blend "Generated Asset Statement â€” PNC â€” 5735 â€” May 2026").

**Root cause (NOT a missing schema):** A complete 26-field bank-statement extraction schema already exists in CatchingDoc â€” but it was registered only under the doc-type string `"Bank Statement"` (singular). The client posts the **actual eFolder bucket title**, which on this loan is `"Bank Statements"` (plural):

| | |
|---|---|
| `data_gathering.py:996` posts | `document_types=[bucket_name]` â†’ `"Bank Statements"` (bucket title) |
| `get_schema("Bank Statement")` | âś… FOUND â€” 26 extraction properties |
| `get_schema("Bank Statements")` | âťŚ 404 not_found |

CatchingDoc matches schemas by **exact DocumentType string**. The plural bucket title located the PDF but missed the schema â†’ the service fell back to raw `document_content` (1 field), so `bank_statement_months` / `statement_period_*` / balances were never populated. `run_pre_checks` then coerced `None` â†’ `0` â†’ false "0 months". (Same class of issue is why the live schema list already duplicates entries like `Lock Confirmation` + `Lock Confirmation Document` and `Evidence of Insurance` Ă—4 â€” one schema per bucket-title variant.)

**Why not just post `"Bank Statement"` (singular) from the client?** The posted string must *also* equal the bucket title for the service to locate the attachment, and bucket titles vary per loan. No single client-side string is safe â€” the robust fix is registering the schema under both names server-side.
